## Idea

The current skills were shaped around a technical project-planning workflow, but the main vault is the true knowledge repository.

The main vault will likely need the same core gates, but with more emphasis on theoretical knowledge extraction, discovery, and comprehension rather than implementation planning.

## Questions

- Should the Project Planning Workflow vault itself be generalized so the skills become broadly useful across knowledge work?
- Or should the main vault get its own `AGENTS.md` that adapts the shared workflow ideas locally while leaving this vault technical-project-focused?
- Which skills need main-vault-specific behavior: `clarify-intent`, `note-manager`, `note-search`, `project-review-sync`, or a separate knowledge extraction skill?
- Should the main vault inherit the same hard gates, or should research and comprehension workflows have different gates?
- How should agents handle the difference between project implementation knowledge and theoretical or conceptual knowledge?

## Current Thinking

- Keeping the current vault as the source for workflow primitives may avoid over-generalizing it too early.
- A main-vault `AGENTS.md` can specialize the same principles for knowledge work: fleeting capture, source material processing, full-note promotion, tagging, and comprehension.
- Shared skills may need wording that supports both technical project workflows and broader knowledge workflows without weakening the gates.

## Open Decisions

- Whether to update installed skills now or wait until the main vault rules are clearer.
- Whether the main vault needs a new skill for source-material extraction and knowledge promotion.
- Whether review/sync is the right skill for extracting Ecommerce learnings, or whether that flow needs a knowledge-collector-style handoff first.
